# Story M9.7 — Canonical Resolution via npx

**Epic:** KZ-M9 (post-release hardening)
**Status:** Ready for Review
**Created:** 2026-04-29
**Source:** Expert-reported bug, session 2026-04-29

---

## Summary

`npx kaizen-dvir@latest update` (and `init`, which routes to update) reported `installed=1.8.0, target=1.8.0` even when the published npm package was at v2.0.1. The update no-opped silently with `same_version`, leaving the expert with no signal that anything had failed. Same path is broken for any npx-based update.

Two compounding defects in the canonical-manifest resolver caused the loop. Story M9.7 fixes both, adds a defensive generator guard, and lands regression coverage.

---

## Root Cause Analysis

### Defect 1 — wrong scope in `CANONICAL_PACKAGE_REL`

`.kaizen-dvir/dvir/update/manifest.js` declared:

```js
const CANONICAL_PACKAGE_REL = path.join('node_modules', '@DanYuzo', 'kaizen-dvir');
```

The package is published unscoped on npm registry. `npm view kaizen-dvir name` returns `kaizen-dvir`; `npm view @DanYuzo/kaizen-dvir` returns 404. The scoped path therefore never exists on a real install — candidate #3 (the most important one for normal `npm i kaizen-dvir` flows) always missed.

### Defect 2 — npx `INSTALL_ROOT` not threaded into the resolver

`bin/kaizen-update.js` line 81:
```js
const INSTALL_ROOT = path.resolve(__dirname, '..');
```

This points at the package root of the running binary — under npx, that's the npx cache extract path (e.g., `~/.npm/_npx/<hash>/node_modules/kaizen-dvir/`). But neither call site of `resolveCanonicalManifest` (lines 810 and 1256) passed it. The resolver had no way to know where the running framework code actually lived.

### Compound failure mode

With both defects in play under npx:

1. Candidate #1 (`opts.canonicalRoot`) — not set in normal CLI use.
2. Candidate #2 (`KAIZEN_CANONICAL_ROOT` env) — not set.
3. Candidate #3 (`<projectRoot>/node_modules/@DanYuzo/kaizen-dvir/`) — wrong scope, doesn't exist.
4. Candidate #4 (`projectRoot` self-checkout fallback) — **does** exist (every initialized project has its own LOCAL `.kaizen-dvir/manifest.json` written by `kaizen-init@1` / `kaizen-update@1`).

The resolver read the user's LOCAL manifest as if it were canonical, returned `installed === target === 1.8.0`, and the update no-opped. From the user's perspective: `npx kaizen-dvir@latest update` did absolutely nothing.

## Acceptance Criteria

- [x] **AC-1 (Fix A):** `CANONICAL_PACKAGE_REL` in `.kaizen-dvir/dvir/update/manifest.js` is unscoped (`node_modules/kaizen-dvir`).
- [x] **AC-2 (Fix B — resolver):** `resolveCanonicalManifest` accepts `opts.installRoot` as the third-priority candidate (after `opts.canonicalRoot` and `KAIZEN_CANONICAL_ROOT` env, before `<projectRoot>/node_modules/kaizen-dvir/`).
- [x] **AC-3 (Fix B — call sites):** Both call sites in `bin/kaizen-update.js` (initial update path at ~810, `--continue` resume path at ~1256) pass `INSTALL_ROOT` as `installRoot`.
- [x] **AC-4 (Fix B — init):** `bin/kaizen-init.js` does NOT call `resolveCanonicalManifest` directly — confirmed via grep, no fix needed there. The init->update routing already flows through the patched code path.
- [x] **AC-5 (Fix C — generator guard):** Resolver rejects any candidate whose `manifest.generator` does not start with `build-canonical-manifest`. Prevents the `projectRoot` self-checkout fallback from ever serving a LOCAL manifest as canonical.
- [x] **AC-6 (Regression coverage):** 4 unit tests in `tests/m9/m9.7/test-canonical-resolution-via-npx.js`:
  - Test 1: unscoped path resolution
  - Test 2: `installRoot` wins over `projectRoot` fallback
  - Test 3: local manifest never returned as canonical (returns `null` instead)
  - Test 4: explicit `canonicalRoot` priority preserved
- [x] **AC-7 (Test fixtures updated):** Existing test fixture builders that plant fake "canonical" manifests now include `generator: 'build-canonical-manifest@1'` so the new guard accepts them.
- [x] **AC-8 (No regressions):** `node tests/run.js` shows zero net regressions vs the pre-M9.7 baseline (1029 pass / 31 fail). After M9.7: 1034 pass / 29 fail (gained 4 from new tests, plus one pre-existing M6.6 channel test now passes due to improved canonical resolution).
